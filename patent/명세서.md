# 특허 명세서

---

## 【발명의 명칭】

에이전트 기반 멀티프로토콜 통신 모니터링 시스템 및 방법

---

## 【기술분야】

본 발명은 산업용 통신 모니터링 시스템에 관한 것으로, 보다 구체적으로는 RS-485, UDP 및 TCP를 포함하는 복수의 통신 프로토콜을 에이전트 기반 아키텍처를 통해 추상화하여 산업용 장치를 통합적으로 모니터링 및 제어하는 시스템과 그 방법에 관한 것이다.

본 발명은 국제 특허 분류(IPC) H04L 43/04 (네트워크 모니터링), G08B 25/00 (경보 시스템), F24F 11/30 (HVAC 제어)에 해당하는 기술 분야에 속한다.

---

## 【배경기술】

산업 현장에서는 제습기, 공조기, 냉동기 등 다수의 설비를 중앙에서 모니터링하고 제어할 필요성이 증가하고 있다. 특히, 대규모 물류 창고, 반도체 클린룸, 식품 저장 시설 등에서는 습도 관리가 제품 품질과 직결되므로, 복수의 제습기를 실시간으로 감시하는 것이 매우 중요하다.

종래의 산업용 장치 모니터링 시스템은 다음과 같은 문제점이 있었다.

첫째, 기존 시스템은 RS-485, Modbus, BACnet 등 단일 통신 프로토콜만을 지원하는 경우가 대부분이다. 그러나 실제 산업 현장에서는 설비의 설치 시기, 제조사, 통신 거리 등에 따라 RS-485, UDP, TCP 등 다양한 프로토콜이 혼재되어 있어, 단일 프로토콜 기반 시스템으로는 전체 설비를 통합적으로 관리할 수 없다. 이로 인해 프로토콜별로 별도의 모니터링 소프트웨어를 운용해야 하며, 운영 비용과 관리 복잡도가 증가한다.

둘째, RS-485 통신 환경에서는 전기적 노이즈, 버스 충돌, 전송 지연 등으로 인해 데이터 패킷이 분할되어 수신되는 현상(패킷 프래그먼테이션)이 빈번하게 발생한다. 종래의 시스템은 이러한 분할 수신 문제에 대한 견고한 처리 메커니즘이 부족하여, 데이터 손실이나 오파싱이 발생할 수 있다.

셋째, 기존 시스템은 장치 상태 변화를 자동으로 감지하는 기능이 미흡하다. 운전 시작/정지, 압축기 가동/정지, 에러 발생/해제 등의 중요한 상태 변화를 운영자가 직접 화면을 주시하며 파악해야 하므로, 대규모 장치 운영 시 상태 변화를 놓칠 위험이 있다.

넷째, 장치 수가 증가함에 따라 주소 지정(addressing)의 효율성이 문제가 된다. 기존의 단순 순차 번호 방식은 그룹별 관리가 어렵고, 대규모 시설에서 장치의 논리적 위치와 물리적 위치를 매핑하기 곤란하다.

따라서, 복수의 통신 프로토콜을 통합 지원하면서도, 패킷 분할 수신을 견고하게 처리하고, 상태 변화를 자동으로 감지하며, 대규모 장치 그룹을 효율적으로 관리할 수 있는 새로운 모니터링 시스템이 요구된다.

## 【해결하고자 하는 과제】

본 발명이 해결하고자 하는 기술적 과제는 다음과 같다.

첫째, RS-485, UDP, TCP 등 이종(異種) 통신 프로토콜을 하나의 공통 인터페이스로 추상화하여, 프로토콜 종류에 관계없이 장치 상태를 통합적으로 모니터링할 수 있는 에이전트 기반 시스템을 제공하는 것이다.

둘째, RS-485 통신 환경에서 발생하는 패킷 분할 수신 문제를 스트림 누적 버퍼와 프레임 패턴 매칭을 통해 견고하게 처리하고, CRC-16 CCITT 무결성 검증을 통해 노이즈 환경에서의 데이터 신뢰성을 보장하는 것이다.

셋째, 폴링 기반 주기적 상태 수집과 이전/현재 상태 비교(diff) 알고리즘을 통해, 운전 시작/정지, 압축기 가동/정지, 에러 발생/해제 등의 상태 변화를 자동으로 감지하고 이벤트를 생성하는 것이다.

넷째, 1바이트 주소 체계에 그룹 식별자(상위 4비트)와 장치 인덱스(하위 4비트)를 패킹하여, 최대 10개 그룹, 그룹당 16대, 총 160대의 장치를 효율적으로 주소 지정할 수 있는 계층적 주소 체계를 제공하는 것이다.

다섯째, 이벤트 발생 시의 단발성 이벤트음과 에러 지속 시의 연속 경고음으로 구성되는 이중 음향 경보 시스템을 통해, 운영자의 상황 인지도를 향상시키는 것이다.

---

## 【과제의 해결 수단】

상기 과제를 해결하기 위한 본 발명의 에이전트 기반 멀티프로토콜 통신 모니터링 시스템은, 복수의 통신 프로토콜에 각각 대응하며 공통 인터페이스를 구현하는 복수의 프로토콜 에이전트(910, 920, 930)와, 상기 복수의 프로토콜 에이전트를 등록하고 프로토콜 종류에 무관한 통합 장치 상태를 상위 계층에 제공하는 프로토콜 추상화 계층(1000, 1300)과, 설정 가능한 주기로 각 장치에 상태 요청 패킷을 전송하는 폴링 엔진(2000)과, 이전 폴링 주기의 장치 상태와 현재 폴링 주기의 장치 상태를 비교하여 상태 변화를 검출하는 상태 변화 감지부(2600)와, 이벤트 발생 시 제1 음향 신호를 출력하고 에러 상태 지속 시 제2 음향 신호를 반복 출력하는 이중 음향 경보부(3330, 3400)와, 통합 장치 상태 및 이벤트를 표시하는 대시보드 표시 계층(1100, 4000)을 포함한다.

또한, 본 발명의 모니터링 방법은, 복수의 프로토콜 에이전트를 통해 이종 통신 프로토콜을 추상화하는 단계와, 계층적 주소 체계를 이용하여 주기적으로 장치 상태를 폴링하는 단계와, 수신 데이터를 스트림 버퍼에 누적하여 완전한 프레임을 추출하는 단계와, CRC-16 검증으로 프레임 무결성을 확인하는 단계와, 이전/현재 상태 비교를 통해 상태 변화를 감지하는 단계와, 이중 음향 경보를 통해 운영자에게 알리는 단계를 포함한다.

## 【발명의 효과】

본 발명에 따르면, 다음과 같은 효과가 달성된다.

첫째, 에이전트 기반 프로토콜 추상화 아키텍처를 통해, RS-485, UDP, TCP 등 이종 프로토콜로 연결된 장치를 하나의 통합 대시보드에서 모니터링할 수 있다. 새로운 통신 프로토콜이 추가되더라도 공통 인터페이스를 구현하는 에이전트만 추가하면 되므로 시스템의 확장성이 우수하다.

둘째, 스트림 누적 버퍼와 프레임 패턴 매칭 알고리즘을 통해, RS-485 통신 환경에서 빈번하게 발생하는 패킷 분할 수신 문제를 견고하게 처리할 수 있다. CRC-16 CCITT 무결성 검증과 결합하여 노이즈 환경에서도 데이터 신뢰성을 보장한다.

셋째, 폴링 기반 상태 비교(diff) 알고리즘을 통해 압축기 가동/정지, 운전 시작/정지, 에러 발생/해제 등의 상태 변화를 자동으로 감지하여 이벤트를 생성하므로, 운영자가 화면을 지속적으로 주시하지 않아도 장애 상황을 파악할 수 있다.

넷째, 1바이트 계층적 주소 체계(그룹 4비트 + 장치 4비트)를 통해 최대 160대의 장치를 효율적으로 관리할 수 있으며, 그룹별 독립 설정과 뷰 필터링이 가능하여 대규모 시설 운영에 적합하다.

다섯째, 이벤트 발생 시 단발성 이벤트음(제1 주파수)과 에러 지속 시 연속 경고음(제2 주파수)으로 구성되는 이중 음향 경보 시스템을 통해, 운영자의 상황 인지도를 향상시킨다. 두 음향 경보는 독립적으로 활성/비활성 설정이 가능하며 설정이 영속적으로 저장된다.

여섯째, Electron 프레임워크의 프로세스 격리 메커니즘(contextBridge)을 활용하여, 하드웨어 직접 접근이 필요한 메인 프로세스와 사용자 인터페이스를 담당하는 렌더러 프로세스 사이에 보안 격리 경계를 설정함으로써, 안전한 하드웨어 접근을 보장한다.

## 【도면의 간단한 설명】

**도 1**은 본 발명의 일 실시예에 따른 에이전트 기반 멀티프로토콜 통신 모니터링 시스템의 전체 구성도이다. RS-485 버스(200)를 통해 복수의 제습기 장치(110)가 연결되고, USB-RS485 변환기(300)를 거쳐 호스트 PC(400)의 모니터링 에이전트(410)에 접속하는 구조를 나타낸다. 향후 UDP/TCP 네트워크(500)를 통한 확장 구조도 점선으로 표시한다.

**도 2**는 본 발명의 소프트웨어 에이전트 아키텍처의 계층별 블록도이다. 하드웨어 계층(600)에서 렌더러 프로세스 계층(1100)까지 6개 계층으로 구성되며, IPC 보안 격리 경계(800)를 통해 프로세스 간 통신이 이루어지는 구조를 나타낸다.

**도 3**은 프로토콜 추상화 계층의 구조도이다. IProtocolAgent 공통 인터페이스(1200)를 RS485Agent(1210), UDPAgent(1220), TCPAgent(1230)가 구현하고, 프로토콜 추상화 계층(1300)이 이들을 집합하여 통합 DeviceStatus(1400)를 출력하는 구조를 나타낸다.

**도 4**는 통신 프레임 구조도이다. RS-485 요청 프레임(1500) 및 상태 응답 프레임(1510)의 바이트 단위 구조와, ID 바이트(1520)의 비트 분할 구조, Data1 바이트(1530)의 비트 필드 구성, 그리고 향후 구현 예정인 UDP 프레임(1600) 및 TCP 프레임(1700)의 구조를 나타낸다.

**도 5**는 데이터 처리 흐름도이다. 폴링 타이머(2000)로부터 요청 패킷 생성(2100), IPC 전송(2200), 응답 수신 및 버퍼 처리(2300), 프레임 추출 및 검증(2400), 상태 파싱 및 갱신(2500), 변화 감지 및 이벤트 발생(2600), 버퍼 정리(2700)까지의 전체 데이터 처리 흐름을 나타낸다.

**도 6**은 이벤트 감지 및 알림 흐름도이다. 상태 비교 입력(3000)으로부터 이벤트 감지 조건(3100) 판단, addEvent 이벤트 생성(3200), 그리고 UI 갱신(3310), 파일 로깅(3320), 이벤트음 재생(3330)의 3가지 병렬 처리 경로와 에러 상태 경고음 루프 제어(3400)를 나타낸다.

**도 7**은 대시보드 UI 구성도이다. 상단 툴바(4100), 장치 제어 테이블(4200), 실시간 이벤트 패널(4300), 그리고 통신설정 모달(4400), 환경설정 모달(4410), 이벤트이력 모달(4420), 꺼짐예약 모달(4430)의 전체 화면 레이아웃을 나타낸다.

## 【발명을 실시하기 위한 구체적인 내용】

이하, 첨부된 도면을 참조하여 본 발명의 바람직한 실시예를 상세히 설명한다. 본 발명의 실시예는 여러 가지 다른 형태로 변형될 수 있으며, 본 발명의 범위가 이하에서 상술하는 실시예에 한정되는 것은 아니다.

### 1. 시스템 전체 구성

도 1을 참조하면, 본 발명의 에이전트 기반 멀티프로토콜 통신 모니터링 시스템은 크게 제습기 장치군(100), RS-485 통신 버스(200), USB-RS485 변환기(300), 호스트 PC(400), 그리고 향후 확장을 위한 UDP/TCP 네트워크(500)로 구성된다.

#### 1.1 제습기 장치군

제습기 장치군(100)은 복수의 제습기 장치(110-1, 110-2, ..., 110-N)를 포함한다. 각 제습기 장치(110)는 RS-485 통신 모듈을 내장하고 있으며, 소정의 프로토콜에 따라 상태 정보를 응답하고 제어 명령을 수신할 수 있다.

각 제습기 장치(110)에는 그룹 식별자(GroupID)와 장치 인덱스(DeviceIndex)가 할당된다. 예를 들어, 도 1에 도시된 바와 같이, 제습기 1(110-1)에는 GroupID=0, Index=0이, 제습기 N(110-N)에는 GroupID=1, Index=0이 할당될 수 있다. 이 주소 체계에 대해서는 후술하는 제4절에서 상세히 설명한다.

각 제습기 장치(110)는 온도 센서, 습도 센서, 증발기 온도 센서를 구비하며, 압축기, 팬 모터 등의 구동부를 포함한다. 장치의 내부 마이크로컨트롤러는 센서 데이터를 수집하고, 호스트 PC(400)로부터의 제어 명령에 따라 구동부를 제어한다.

#### 1.2 RS-485 통신 버스

RS-485 통신 버스(200)는 2선식 차동 신호(differential signal) 방식의 반이중(half-duplex) 통신 버스이다. 최대 1,200m의 통신 거리를 지원하며, 하나의 버스에 최대 32개의 노드를 연결할 수 있다. 본 발명에서는 그룹 단위로 RS-485 버스를 분리하여 구성할 수 있으며, 이를 통해 그룹당 최대 16대의 장치를 수용한다.

통신 파라미터는 다음과 같이 설정된다:
- 전송 속도(Baud Rate): 9,600 / 19,200 / 38,400 / 57,600 / 115,200 bps 중 선택
- 데이터 비트: 8비트
- 패리티: None
- 정지 비트: 1비트

상기 통신 파라미터는 호스트 PC(400)의 통신 설정 인터페이스(4400)를 통해 사용자가 변경할 수 있으며, 변경된 설정은 로컬 스토리지에 영속적으로 저장된다.

#### 1.3 USB-RS485 변환기

USB-RS485 변환기(300)는 호스트 PC(400)의 USB 포트와 RS-485 버스(200) 사이의 물리 계층 변환을 담당한다. 변환기(300)는 호스트 PC(400)의 운영체제에 의해 가상 직렬 포트(Virtual COM Port)로 인식되며, 이를 통해 소프트웨어가 직렬 포트 API를 사용하여 RS-485 통신을 수행한다.

#### 1.4 호스트 PC 및 모니터링 에이전트

호스트 PC(400)에는 모니터링 에이전트(410) 소프트웨어가 설치된다. 모니터링 에이전트(410)는 Electron 프레임워크 기반의 데스크톱 애플리케이션으로 구현되며, 그 상세한 소프트웨어 아키텍처는 도 2를 참조하여 후술한다.

#### 1.5 네트워크 확장 구조

도 1에 점선으로 도시된 바와 같이, 본 발명의 시스템은 UDP/TCP 네트워크(500)를 통해 네트워크 장치군(510)을 추가적으로 연결할 수 있는 확장 구조를 갖는다. UDP 장치(510-UDP)는 비연결형 데이터그램 방식으로, TCP 장치(510-TCP)는 연결 지향 스트림 방식으로 통신한다. 이러한 네트워크 확장은 본 발명의 프로토콜 추상화 계층(1000, 1300)의 설계에 의해 기존 RS-485 연결과 동일한 방식으로 관리될 수 있다.

### 2. 에이전트 아키텍처

도 2를 참조하면, 본 발명의 모니터링 에이전트(410)의 소프트웨어 아키텍처는 6개의 계층으로 구성된다. 최하위 계층부터 최상위 계층까지의 구성은 다음과 같다.

#### 2.1 하드웨어 계층 (Layer 1, 참조번호 600)

하드웨어 계층(600)은 RS-485 포트(610), USB 포트(620), 네트워크 인터페이스(630)를 포함한다. RS-485 포트(610)는 USB-RS485 변환기(300)를 통해 호스트 PC(400)에 가상 직렬 포트로 노출된다. 네트워크 인터페이스(630)는 향후 UDP/TCP 통신을 위해 사용된다.

#### 2.2 Electron 메인 프로세스 계층 (Layer 2, 참조번호 700)

Electron 메인 프로세스 계층(700)은 Node.js 런타임 환경에서 동작하며, 하드웨어에 직접 접근할 수 있는 유일한 계층이다. 본 계층은 다음의 모듈을 포함한다:

- **SerialPort 모듈(710)**: node-serialport 라이브러리를 사용하여 가상 직렬 포트를 제어한다. 포트 열기, 닫기, 데이터 쓰기, 데이터 수신 이벤트 처리를 담당한다. 열려 있는 포트를 Map 자료구조로 관리하여 동일 포트의 중복 열기를 방지하며, 애플리케이션 종료 시 모든 열린 포트를 자동으로 닫는다.
- **UDP Socket(720)**: 향후 구현 예정이며, Node.js의 dgram 모듈을 사용하여 UDP 데이터그램 통신을 수행한다.
- **TCP Socket(730)**: 향후 구현 예정이며, Node.js의 net 모듈을 사용하여 TCP 스트림 통신을 수행한다.

메인 프로세스 계층(700)은 다음의 IPC(Inter-Process Communication) 채널을 통해 상위 계층과 통신한다:

| IPC 채널 | 방향 | 용도 |
|---|---|---|
| serial:listPorts | 렌더러→메인 | 연결 가능 포트 목록 조회 |
| serial:openPort | 렌더러→메인 | 포트 열기 (baudRate 설정 포함) |
| serial:closePort | 렌더러→메인 | 포트 닫기 |
| serial:write | 렌더러→메인 | 바이너리 데이터 송신 |
| serial:data | 메인→렌더러 | 수신 데이터 푸시 (base64 인코딩) |
| log:appendLine | 렌더러→메인 | 이벤트 로그 파일 기록 |

수신 데이터의 전달 과정에서, Node.js Buffer 형태의 바이너리 데이터는 base64 문자열로 직렬화되어 IPC를 통해 전달되며, 렌더러 프로세스에서 다시 Uint8Array로 역직렬화된다. 이 변환 과정은 IPC 경계에서의 바이너리-텍스트 안전성을 보장한다.

#### 2.3 IPC 보안 격리 경계 (Layer 3, 참조번호 800)

IPC 보안 격리 경계(800)는 Electron 프레임워크의 contextBridge 메커니즘을 활용하여 구현된다. 본 계층의 핵심 보안 설계는 다음과 같다:

- **컨텍스트 격리(Context Isolation)**: `contextIsolation: true` 설정을 통해, 렌더러 프로세스의 JavaScript 실행 컨텍스트와 메인 프로세스의 실행 컨텍스트를 완전히 분리한다.
- **노드 통합 비활성화**: `nodeIntegration: false` 설정을 통해, 렌더러 프로세스에서 Node.js API에 직접 접근하는 것을 차단한다.
- **최소 권한 원칙**: preload 스크립트를 통해 사전 정의된 5개의 serialApi 메서드(listPorts, openPort, closePort, write, onData)와 1개의 logApi 메서드(appendLine)만을 `window` 객체에 노출한다. 이를 통해 렌더러 프로세스는 허용된 IPC 채널만을 사용할 수 있다.

이러한 보안 설계는 렌더러 프로세스가 악의적인 코드에 의해 침해되더라도, 하드웨어 접근이나 파일 시스템 접근이 사전 정의된 범위 내로 제한되는 것을 보장한다.

#### 2.4 프로토콜 에이전트 계층 (Layer 4, 참조번호 900)

프로토콜 에이전트 계층(900)은 각 통신 프로토콜에 대응하는 에이전트를 포함한다. 도 3을 참조하면, 각 에이전트는 공통 인터페이스 IProtocolAgent(1200)를 구현한다.

- **RS485 에이전트(910)**: 현재 구현된 에이전트로서, SerialPort 모듈(710)을 사용하여 RS-485 프로토콜에 따른 패킷 송수신을 수행한다. poll() 메서드에서 상태 요청 패킷을 생성하여 전송하고, onData() 콜백을 통해 수신 데이터를 처리한다.
- **UDP 에이전트(920)**: 향후 구현 예정이며, dgram Socket을 사용하여 UDP 데이터그램을 송수신한다.
- **TCP 에이전트(930)**: 향후 구현 예정이며, net.Socket을 사용하여 TCP 스트림을 송수신한다.

#### 2.5 프로토콜 추상화 계층 (Layer 5, 참조번호 1000)

프로토콜 추상화 계층(1000)은 복수의 프로토콜 에이전트(910, 920, 930)를 등록하고, 프로토콜 종류에 무관한 통합 장치 상태(1400)를 상위 계층에 제공한다. 본 계층은 이전 상태와 현재 상태의 비교(diff), 이벤트 발생 로직, hasAnyDeviceError 연산 등을 포함한다.

#### 2.6 렌더러 프로세스 계층 (Layer 6, 참조번호 1100)

렌더러 프로세스 계층(1100)은 React 기반 SPA(Single Page Application)로 구현되며, 장치 제어 패널, 이벤트 로그 패널, 통신 설정 모달, 환경 설정 모달 등의 UI 컴포넌트를 포함한다. 상세한 UI 구성은 도 7을 참조하여 후술한다.

### 3. 프로토콜 추상화 계층 상세

도 3을 참조하면, 프로토콜 추상화 계층의 상세 구조는 다음과 같다.

#### 3.1 IProtocolAgent 공통 인터페이스

IProtocolAgent 공통 인터페이스(1200)는 모든 프로토콜 에이전트가 구현해야 하는 6개의 메서드를 정의한다:

- `connect(config: ConnectionConfig): Promise<void>` - 통신 연결을 수립한다. RS-485의 경우 포트 열기, UDP의 경우 소켓 바인딩, TCP의 경우 서버 연결에 해당한다.
- `disconnect(): Promise<void>` - 통신 연결을 해제한다.
- `poll(deviceList: DeviceInfo[]): void` - 지정된 장치 목록에 대해 상태 요청 패킷을 전송한다.
- `sendCommand(groupId: number, index: number, cmd: CommandType): void` - 특정 장치에 제어 명령을 전송한다.
- `onData(callback: (status: DeviceStatus) => void): void` - 수신 데이터 처리 콜백을 등록한다.
- `getProtocolType(): ProtocolType` - 에이전트의 프로토콜 유형("RS485", "UDP", "TCP")을 반환한다.

이 공통 인터페이스를 통해, 상위 계층은 실제 통신 프로토콜의 세부 사항을 알 필요 없이 동일한 방식으로 모든 에이전트와 상호작용할 수 있다.

#### 3.2 RS485Agent 구현체

RS485Agent(1210)는 IProtocolAgent(1200)를 구현하는 현재 활성화된 에이전트이다. 내부적으로 SerialPort 인스턴스, 수신 버퍼(rxBuffer), 폴링 타이머(pollTimer)를 관리한다.

RS485Agent(1210)의 `connect()` 메서드는 IPC를 통해 SerialPort.open()을 호출하고, `poll()` 메서드는 장치 목록을 순회하며 buildStatusRequest() 함수로 패킷을 생성하여 writeBinary()로 전송한다. `onData()` 메서드는 수신 바이트를 rxBuffer에 누적하고, extractStatusFrames() 함수를 통해 완전한 프레임을 추출하여 파싱한다.

#### 3.3 프로토콜 추상화 계층 집합체

프로토콜 추상화 계층(1300)은 Map<ProtocolType, IProtocolAgent> 자료구조로 복수의 에이전트를 관리한다. registerAgent() 메서드로 에이전트를 등록하고, connectAll() 메서드로 모든 에이전트의 연결을 수립하며, startPolling() 메서드로 모든 에이전트의 폴링을 시작한다. 각 에이전트로부터 수신된 DeviceStatus는 프로토콜 종류에 관계없이 통합 DeviceStatus(1400) 인터페이스로 정규화되어 상위 계층에 제공된다.

#### 3.4 통합 DeviceStatus 출력

통합 DeviceStatus(1400)는 프로토콜 종류에 무관하게 장치 상태를 표현하는 공통 데이터 구조체이다. flatIndex(장치 고유 인덱스), groupId(그룹 ID), deviceIndex(장치 번호), location(설치 위치), runStart(운전 상태), compressor(압축기 상태), currentHumi(현재 습도), setHumi(설정 습도), temperature(온도), fanStrong(풍량), opMode(운전 모드), reserveTime(예약 시간), errorCode(에러 코드), protocolType(프로토콜 종류), timestamp(수신 시각) 등의 필드를 포함한다.

### 4. 통신 프레임 구조

도 4를 참조하면, 본 발명의 통신 프레임 구조는 다음과 같다.

#### 4.1 ID 바이트 인코딩 체계

ID 바이트(1520)는 1바이트(8비트) 크기이며, 상위 4비트(bit[7:4])에 그룹 식별자(GroupID)를, 하위 4비트(bit[3:0])에 장치 인덱스(DeviceIndex)를 패킹하는 구조를 갖는다.

```
ID 바이트 = (GroupID & 0x0F) << 4 | (DeviceIndex & 0x0F)
```

GroupID는 0~9의 범위(최대 10개 그룹)를 가지며, DeviceIndex는 0~15의 범위(그룹당 최대 16대)를 가진다. 따라서, 1바이트의 주소 공간으로 최대 10 × 16 = 160대의 장치를 고유하게 식별할 수 있다.

예를 들어, GroupID=2, DeviceIndex=3인 장치의 ID 바이트는:
```
ID = (0x02 << 4) | 0x03 = 0x20 | 0x03 = 0x23
```

수신 측에서의 역분해(디코딩)는 다음과 같다:
```
GroupID     = (ID >> 4) & 0x0F  // 상위 니블 추출
DeviceIndex = ID & 0x0F         // 하위 니블 추출
```

이러한 계층적 주소 체계는 물리적 위치(그룹)와 장치 순번을 1바이트에 함축적으로 표현함으로써, 패킷 오버헤드를 최소화하면서도 직관적인 그룹별 장치 관리를 가능하게 한다.

#### 4.2 RS-485 요청 프레임

RS-485 요청 프레임(1500)은 다음의 공통 구조를 갖는다:

```
[CMD(1B)][LEN(1B)][ID(1B)][DATA(0~2B)][CRC16(2B)]
```

여기서 CMD는 명령 코드, LEN은 ID 바이트를 포함한 데이터 길이(ID 1바이트 + DATA 바이트 수), ID는 상술한 그룹-장치 인코딩 바이트, DATA는 명령별 파라미터, CRC16은 CRC-16 CCITT 검증 코드이다.

본 발명에서 정의하는 6종의 명령 코드와 각 패킷 구조는 다음과 같다:

**(1) 상태 요청 (CMD = 0x91)**

```
[0x91][0x01][ID][CRC_H][CRC_L]  → 총 5바이트
```

LEN = 0x01 (ID 1바이트, DATA 없음). 지정된 장치에 현재 상태를 요청한다.

**(2) 운전 ON/OFF (CMD = 0x92)**

```
[0x92][0x02][ID][RUN][CRC_H][CRC_L]  → 총 6바이트
```

LEN = 0x02 (ID 1바이트 + DATA 1바이트). RUN = 0x01(운전 시작) 또는 0x00(운전 정지).

**(3) 운전 모드 변경 (CMD = 0x93)**

```
[0x93][0x02][ID][MODE][CRC_H][CRC_L]  → 총 6바이트
```

LEN = 0x02. MODE = 0x00(송풍), 0x01(자동), 0x02(제습).

**(4) 풍량 설정 (CMD = 0x94)**

```
[0x94][0x02][ID][SPEED][CRC_H][CRC_L]  → 총 6바이트
```

LEN = 0x02. SPEED = 0x00(약풍), 0x01(강풍).

**(5) 희망 습도 설정 (CMD = 0x95)**

```
[0x95][0x02][ID][HUMI][CRC_H][CRC_L]  → 총 6바이트
```

LEN = 0x02. HUMI = 10~90 (% 단위).

**(6) 꺼짐 예약 (CMD = 0x96)**

```
[0x96][0x03][ID][RSV_H][RSV_L][CRC_H][CRC_L]  → 총 7바이트
```

LEN = 0x03 (ID 1바이트 + DATA 2바이트). RSV_H의 bit7은 예약 활성(1)/비활성(0) 플래그이고, bit[6:0]은 시간(0~24). RSV_L은 분(0~59).

#### 4.3 RS-485 상태 응답 프레임

상태 응답 프레임(1510)은 CMD = 0xC1이며, 고정 길이 18바이트이다. 그 상세 구조는 다음과 같다:

| 오프셋 | 필드 | 크기 | 설명 |
|---|---|---|---|
| 0 | CMD | 1B | 0xC1 (상태 응답 코드) |
| 1 | LEN | 1B | 0x0D (= 13, ID 1B + 데이터 12B) |
| 2 | ID | 1B | 그룹ID[7:4] + 장치인덱스[3:0] |
| 3 | Data1 | 1B | 운전 상태 비트 필드 (1530) |
| 4 | HumiSet | 1B | 설정 습도 (%) |
| 5 | RsvTimeH | 1B | 꺼짐예약 시간 (bit7=활성, bit[6:0]=시) |
| 6 | RsvTimeL | 1B | 꺼짐예약 분 (0~59) |
| 7 | Temp | 1B | 온도 raw (실제 °C = raw - 40, 0xFF=에러) |
| 8 | Humi | 1B | 현재 습도 (%, 255=에러) |
| 9 | EvaT2 | 1B | 증발기 온도 2 |
| 10 | EvaT1 | 1B | 증발기 온도 1 |
| 11 | Switch | 1B | 스위치 상태 |
| 12 | Output | 1B | 출력 상태 (bit0=압축기 가동) |
| 13 | Error | 1B | 에러 비트 필드 |
| 14-15 | CRC16 | 2B | CRC-16 CCITT (빅엔디안) |

#### 4.4 Data1 비트 필드

Data1 바이트(1530, 오프셋 3)의 비트 필드 구성은 다음과 같다:

| 비트 | 필드명 | 마스크 | 값 의미 |
|---|---|---|---|
| bit[7] | COMM_ERROR | 0x80 | 0=정상, 1=E9 통신 에러 |
| bit[6] | (예약) | 0x40 | - |
| bit[5] | RUN_START | 0x20 | 0=정지, 1=운전 중 |
| bit[4] | FAN_STRONG | 0x10 | 0=약풍, 1=강풍 |
| bit[3:2] | (예약) | 0x0C | - |
| bit[1:0] | OP_MODE | 0x03 | 0=송풍, 1=자동, 2=제습 |

#### 4.5 에러 코드 체계

Error 바이트(오프셋 13)의 각 비트는 독립적인 에러 상태를 나타낸다:

| 비트 마스크 | 에러 코드 | 설명 |
|---|---|---|
| 0x01 | E1 | 고압 (High Pressure) |
| 0x02 | E2 | 저압 (Low Pressure) |
| 0x04 | E3 | 에바동결 (Evaporator Freeze) |
| 0x08 | E4 | 에바온도 (Evaporator Temperature) |
| 0x10 | E5 | 토출온도 (Discharge Temperature) |
| 0x20 | E6 | 모듈온도 (Module Temperature) |
| 0x40 | E7 | 온습도 (Humidity Sensor) |
| 0x80 | E8 | 만수위 (Tank Full) |

추가로, Data1 바이트의 bit7이 1인 경우 E9 통신 에러를 나타낸다. 복수의 에러 비트가 동시에 설정될 수 있으며, 이 경우 우선순위가 높은(비트 번호가 낮은) 에러가 대표 에러로 표시되되, 개별 에러 메시지 목록도 함께 제공된다.

#### 4.6 CRC-16 CCITT 무결성 검증

모든 프레임(요청 및 응답)에는 CRC-16 CCITT 검증 코드가 부가된다. 계산 알고리즘의 사양은 다음과 같다:

- **다항식(Polynomial)**: 0x1021 (x^16 + x^12 + x^5 + 1)
- **초기값(Initial Value)**: 0xFFFF
- **계산 범위**: CMD 바이트부터 마지막 데이터 바이트까지 (CRC 필드 자체는 제외)
- **바이트 순서**: 빅엔디안 (상위 바이트 먼저, 하위 바이트 나중)

CRC 계산의 의사 코드는 다음과 같다:

```
CRC ← 0xFFFF
for each byte B in [CMD, LEN, ID, DATA...]:
    CRC ← CRC XOR (B << 8)
    for j ← 0 to 7:
        if (CRC AND 0x8000) ≠ 0:
            CRC ← ((CRC << 1) XOR 0x1021) AND 0xFFFF
        else:
            CRC ← (CRC << 1) AND 0xFFFF
// 패킷 끝에 [CRC >> 8, CRC & 0xFF] 를 빅엔디안으로 부가
```

수신 측에서의 CRC 검증은, 수신된 프레임의 CRC 필드 직전까지의 바이트에 대해 동일한 알고리즘으로 CRC를 계산하고, 계산된 값과 수신된 CRC 값을 비교하여 일치 여부를 확인한다. 불일치 시 해당 프레임은 손상된 것으로 간주하여 폐기한다.

#### 4.7 UDP 프레임 (향후 구현)

UDP 프레임(1600)은 비연결형 데이터그램 특성에 맞추어 헤더(4B, 0xAA555566)와 푸터(2B, 0x55AA)로 프레임 경계를 명시하고, 시퀀스 번호(SeqNo, 2B)를 포함하여 패킷 순서와 손실을 관리한다. CMD, LEN, ID, DATA 영역은 RS-485 프레임과 동일한 구조를 공유하며, CRC-16 검증도 동일한 알고리즘을 사용한다.

#### 4.8 TCP 프레임 (향후 구현)

TCP 프레임(1700)은 스트림 지향 프로토콜의 특성에 맞추어 STX(0x02)와 ETX(0x03) 구분자로 프레임 경계를 표시하고, 2바이트 Length 필드(빅엔디안)로 전체 프레임 길이를 명시한다. TCP의 신뢰성 있는 전달 특성에도 불구하고, 애플리케이션 레벨의 데이터 무결성 확인을 위해 CRC-16 검증을 유지한다.

### 5. 데이터 처리 흐름

도 5를 참조하면, 폴링 기반 데이터 처리의 전체 흐름은 다음과 같다.

#### 5.1 폴링 엔진

폴링 타이머(2000)는 소정의 주기(기본값 1,500밀리초)로 폴링 사이클을 반복 실행한다. 폴링 사이클이 시작되면, 현재 선택된 그룹에 속한 모든 장치(deviceList)를 순회하며, 각 장치에 대해 상태 요청 패킷을 순차적으로 전송한다.

폴링은 통신이 연결된 상태에서만 활성화되며, 연결이 해제되면 폴링 타이머가 자동으로 중지된다. 또한, 장치 목록이 변경되면(그룹 전환 또는 환경 설정 변경 시) 폴링이 재시작되어 새로운 장치 목록에 맞추어 동작한다.

#### 5.2 요청 패킷 생성 (참조번호 2100)

각 장치에 대해, buildStatusRequest(groupId, deviceIndex) 함수가 호출되어 상태 요청 패킷이 생성된다. 이 함수는 다음의 과정을 수행한다:

1. CMD 바이트(0x91)와 LEN 바이트(0x01)를 설정한다.
2. makeIdByte() 함수를 호출하여 GroupID와 DeviceIndex를 1바이트로 패킹한다.
3. 3바이트 페이로드([0x91, 0x01, ID])에 대해 appendCrc16() 함수를 호출하여 CRC-16을 계산하고 2바이트를 부가한다.
4. 총 5바이트의 완성된 패킷을 반환한다.

#### 5.3 IPC 전송 (참조번호 2200)

생성된 패킷(Uint8Array)은 base64 문자열로 인코딩된 후, IPC를 통해 메인 프로세스로 전달된다. 메인 프로세스의 SerialPort 모듈(710)은 이를 다시 Buffer로 변환하여 직렬 포트에 기록한다. 전송 성공 시 TX LED 인디케이터가 320밀리초 동안 점등된다.

#### 5.4 응답 수신 및 버퍼 처리 (참조번호 2300)

장치로부터의 응답 데이터는 SerialPort의 'data' 이벤트를 통해 수신된다. 수신된 데이터 청크(chunk)는 base64로 인코딩되어 IPC를 통해 렌더러 프로세스로 전달된다. 렌더러 프로세스에서는 base64를 Uint8Array로 디코딩한 후, rxBuffer(수신 누적 버퍼)에 추가한다.

**스트림 누적 버퍼 알고리즘**은 다음과 같이 동작한다:

```
// 기존 버퍼와 새로 수신된 데이터를 연결
prev ← rxBufferRef.current
next ← new Uint8Array(prev.length + chunk.length)
next.set(prev)
next.set(chunk, prev.length)

// 완전한 프레임 추출 시도
frames ← extractStatusFrames(next)
maxEnd ← max(frame.end for frame in frames)

// 처리된 바이트 이후의 잔여 데이터만 보존
rxBufferRef.current ← (maxEnd > 0) ? next.slice(maxEnd) : next
```

이 알고리즘은 RS-485 통신에서 빈번하게 발생하는 패킷 분할 수신(partial receive) 문제를 해결한다. 완전한 프레임이 형성될 때까지 데이터를 누적하고, 프레임 추출 후 소비된 바이트를 제거하여 메모리를 효율적으로 관리한다.

#### 5.5 프레임 추출 및 검증 (참조번호 2400)

extractStatusFrames() 함수는 누적 버퍼에서 완전한 상태 응답 프레임을 추출한다. 알고리즘은 다음과 같다:

```
searchStart ← 0
results ← []
while true:
    // 1. 프레임 시작 패턴 검색: CMD=0xC1, LEN=0x0D
    start ← findStatusFrameStart(buffer, searchStart)
    if start < 0: break

    // 2. 완전한 18바이트 프레임 존재 확인
    end ← start + 18
    if end > buffer.length: break  // 불완전 프레임 → 다음 수신 대기

    // 3. 프레임 슬라이스 및 CRC-16 검증
    frame ← buffer[start .. end]
    if verifyCrc16(frame):
        parsed ← parseStatusFrame(frame)
        if parsed ≠ null:
            results.append({parsed, start, end})

    // 4. 다음 프레임 검색 위치 이동
    searchStart ← end

return results
```

프레임 시작 감지 조건은 두 가지이다: (1) 해당 바이트가 상태 응답 코드 0xC1이고, (2) 다음 바이트가 데이터 길이 0x0D(= 13, ID 1바이트 + 데이터 12바이트)인 경우. 이 두 조건의 결합은 프레임 시작 위치를 높은 확률로 정확하게 식별한다.

CRC-16 검증은 프레임의 오프셋 0~13(CMD부터 Error까지 14바이트)에 대해 CRC를 계산하고, 오프셋 14~15의 수신된 CRC 값과 비교한다. 검증에 실패한 프레임은 폐기된다.

#### 5.6 상태 파싱 및 갱신 (참조번호 2500)

CRC 검증을 통과한 프레임은 parseStatusFrame() 함수에 의해 파싱된다. 파싱 결과는 ParsedStatus 구조체로 반환되며, 원시 값(raw values)과 해석된 값(interpreted values)을 모두 포함한다.

주요 해석 규칙은 다음과 같다:

- **온도**: raw 값에서 오프셋 40을 감산한다 (tempC = raw - 40). raw = 0xFF인 경우 센서 에러로 null을 반환한다.
- **습도**: raw = 255인 경우 센서 에러로 null을 반환한다.
- **압축기 상태**: Output 바이트의 bit0 (마스크 0x01)으로 판정한다.
- **운전 상태**: Data1 바이트의 bit5 (마스크 0x20)로 판정한다.
- **운전 모드**: Data1 바이트의 bit[1:0] (마스크 0x03)으로 판정한다. 0=송풍, 1=자동, 2=제습.
- **풍량**: Data1 바이트의 bit4 (마스크 0x10)로 판정한다. 0=약풍, 1=강풍.
- **꺼짐예약**: RsvTimeH 바이트의 bit7 (마스크 0x80)로 활성 여부를 판정하고, bit[6:0]을 시간, RsvTimeL을 분으로 해석한다.
- **에러 상태**: Error 바이트의 각 비트를 검사하여 E1~E8 에러를 판정하고, Data1 바이트의 bit7로 E9 통신 에러를 판정한다. 복수 에러 시 우선순위가 높은 에러를 대표 에러로 선택한다.

파싱된 상태는 응답 프레임의 ID 바이트를 역분해하여 해당 장치의 flatIndex를 결정한 후, deviceRows 배열의 해당 인덱스에 갱신된다.

#### 5.7 상태 변화 감지 및 이벤트 발생 (참조번호 2600)

상태 변화 감지부(2600)는 Map<flatIndex, ParsedStatus> 구조의 이전 상태 저장소(previousRef)를 유지한다. 각 폴링 응답이 파싱될 때마다, 해당 장치의 이전 상태와 현재 상태를 비교하여 다음 세 가지 유형의 변화를 감지한다:

**(1) 압축기 가동 상태 변화**
```
if prev.compressorOn ≠ current.compressorOn:
    이벤트 발생: "압축기 가동" 또는 "압축기 정지"
```

**(2) 운전 시작/정지 변화**
```
if prev.runStart ≠ current.runStart:
    이벤트 발생: "운전 시작" 또는 "운전 정지"
```

**(3) 에러 상태 변화**
```
if prev.error ≠ current.error OR (prev.data1 & 0x80) ≠ (current.data1 & 0x80):
    if current.error ≠ 0 OR (current.data1 & 0x80) ≠ 0:
        이벤트 발생: "에러: {에러상태}"
```

변화가 감지되면, onEvent() 콜백을 통해 이벤트가 생성되고, 이벤트 처리 파이프라인(도 6)으로 전달된다. 비교 후에는 현재 상태가 이전 상태 저장소에 저장되어 다음 폴링 주기의 비교 기준이 된다.

#### 5.8 버퍼 정리 및 순환 (참조번호 2700)

프레임 추출이 완료된 후, 처리된 마지막 프레임의 종료 위치(maxEnd)까지의 바이트는 rxBuffer에서 제거된다. 미처 완전한 프레임을 구성하지 못한 잔여 바이트는 보존되어 다음 수신 데이터와 결합된다. 이 슬라이딩 윈도우 방식은 버퍼 크기를 최소한으로 유지하면서도 분할 수신 문제를 해결한다.

### 6. 이벤트 감지 및 알림 시스템

도 6을 참조하면, 이벤트 감지 및 알림 시스템의 상세 흐름은 다음과 같다.

#### 6.1 이벤트 감지 조건 (참조번호 3100)

상태 비교 입력(3000)은 previousStatus Map과 currentParsedStatus를 포함한다. 이벤트 감지 조건(3100)에서는 세 가지 비교가 수행된다:

- 압축기 상태 비교: prev.compressorOn ≠ cur.compressorOn
- 운전 상태 비교: prev.runStart ≠ cur.runStart
- 에러 코드 비교: prev.error ≠ cur.error 또는 통신 에러 비트 변화

이 중 어느 하나라도 변화가 감지되면(OR 조건) 이벤트가 발생한다.

#### 6.2 이벤트 생성 (참조번호 3200)

addEvent() 함수는 다음의 정보를 포함하는 이벤트 레코드를 생성한다:

- **time**: "YYYY-MM-DD HH:mm:ss" 형식의 타임스탬프
- **location**: 해당 장치의 설치 위치명 (환경 설정에서 지정)
- **message**: 이벤트 내용 ("운전 시작", "압축기 가동", "에러: E1 고압" 등)
- **id**: 장치 ID (1-indexed 표시용)

생성된 이벤트는 다음 세 가지 경로로 병렬 처리된다.

#### 6.3 UI 갱신 경로 (참조번호 3310)

이벤트 레코드는 eventRows 배열의 선두에 추가(prepend)된다. React 상태 관리에 의해 이벤트 목록 UI가 자동으로 갱신되며, 가장 최근 이벤트가 목록 최상단에 표시된다.

#### 6.4 파일 로깅 경로 (참조번호 3320)

이벤트 레코드는 TSV(Tab-Separated Values) 형식으로 변환되어 로그 파일에 추가 기록된다. 형식은 다음과 같다:

```
{시간}\t{장소명}\t{장치ID}\t{메시지}\n
```

파일 기록은 IPC를 통해 메인 프로세스의 fs.appendFileSync()로 수행되며, Electron 사용자 데이터 디렉토리에 event.log 파일로 저장된다. appendFileSync의 동기식 기록은 로그 누락을 방지한다.

#### 6.5 이벤트음 재생 경로 (참조번호 3330)

이벤트 발생 시 단발성 이벤트음이 재생된다. Web Audio API의 AudioContext와 OscillatorNode를 사용하며, 사양은 다음과 같다:

- **주파수**: 800 Hz (제1 주파수)
- **지속 시간**: 100 밀리초
- **볼륨**: gain.value = 0.1 (저볼륨)
- **재생 방식**: 매 이벤트마다 새로운 AudioContext를 생성하여 단음을 재생한 후 즉시 해제한다.

이벤트음은 사용자 설정에 의해 활성/비활성을 전환할 수 있으며, 설정 상태는 localStorage에 키 "eventSoundEnabled"로 영속적으로 저장된다. 기본값은 활성(true)이다.

#### 6.6 연속 경고음 루프 제어 (참조번호 3400)

이벤트 발생 여부와 별개로, 시스템은 hasAnyDeviceError 상태를 지속적으로 감시한다. 이 값은 deviceRows 배열의 모든 장치 에러 상태를 검사하여, 하나라도 에러 상태인 장치가 있으면 true가 된다.

hasAnyDeviceError가 false에서 true로 전환되면, 연속 경고음 루프가 시작된다:

- **주파수**: 600 Hz (제2 주파수, 이벤트음과 구별)
- **지속 시간**: 150 밀리초
- **반복 주기**: 1,000 밀리초 (1초 간격)
- **AudioContext 관리**: 경고음 루프 전체에서 하나의 AudioContext를 재사용하여 리소스를 절약한다.

hasAnyDeviceError가 true에서 false로 전환되면(모든 에러 해소), 경고음 루프가 즉시 중지된다:

```
clearInterval(warningIntervalRef)
warningAudioContextRef.close()
warningAudioContextRef ← null
```

경고음도 사용자 설정에 의해 독립적으로 활성/비활성을 전환할 수 있으며, 설정 상태는 localStorage에 키 "warningSoundEnabled"로 저장된다. 기본값은 비활성(false)이다.

이 이중 음향 경보 아키텍처의 설계 근거는 다음과 같다:
- **이벤트음** (800 Hz, 단발): 상태 변화 발생 시 즉시 운영자의 주의를 환기한다. 빈번한 상태 변화에도 짧은 단음이므로 방해가 되지 않는다.
- **경고음** (600 Hz, 연속): 에러 상태가 해소될 때까지 지속적으로 경고하여, 운영자가 자리를 비운 경우에도 에러 상황을 인지할 수 있게 한다.
- 두 음향의 주파수를 다르게 설정(800 Hz vs 600 Hz)함으로써, 운영자가 청각적으로 이벤트 발생과 에러 지속을 구분할 수 있다.

### 7. 대시보드 UI

도 7을 참조하면, 대시보드 UI(4000)의 구성은 다음과 같다.

#### 7.1 상단 툴바 (참조번호 4100)

상단 툴바(4100)는 다음의 구성 요소를 포함한다:

- **그룹 선택 드롭다운**: 0~9번 그룹 중 하나를 선택한다. 선택된 그룹의 장치만 메인 테이블에 표시되고 폴링 대상이 된다. 선택 상태는 localStorage에 영속적으로 저장되어 앱 재시작 후에도 유지된다.
- **전체 전원 ON/OFF 버튼**: 선택된 그룹의 모든 장치에 대해 일괄 운전 시작/정지 명령을 전송한다.
- **환경설정 버튼**: 환경설정 모달(4410)을 연다. 그룹별 장치 수, 장소명 등을 설정한다.
- **통신설정 버튼**: 통신설정 모달(4400)을 연다. 포트, 보드레이트 등을 설정한다.
- **연결/끊기 버튼**: 직렬 포트 연결 상태를 토글한다.
- **TX/RX LED 인디케이터**: 데이터 송신(TX) 및 수신(RX) 시 각각 320밀리초 동안 점등되어 통신 활동을 시각적으로 표시한다.

#### 7.2 장치 제어 테이블 (참조번호 4200)

장치 제어 테이블(4200)은 11개 컬럼으로 구성된 테이블 형태이다:

| 컬럼 | 내용 | 인터랙션 |
|---|---|---|
| ID | 1-indexed 장치 번호 | 표시 전용 |
| 장소 | 설치 위치명 | 표시 전용 |
| 운전 시작 | ON/OFF 토글 버튼 | 클릭 → sendRunOnOff() |
| 현재 습도 | 현재 습도(%) | 표시 전용, 운전 중 강조 |
| 희망 습도 | 설정 습도(%) | 표시 전용 |
| 습도 조절 | ↑/↓ 스텝퍼 | 클릭 → sendHumiditySet() |
| 압축기 상태 | 색상 점등 (녹색=가동, 회색=정지) | 표시 전용 |
| 운전 모드 | 송풍/제습/자동 토글 | 클릭 → sendRunMode() |
| 송풍 모드 | 강풍/약풍 토글 | 클릭 → sendFanSpeed() |
| 꺼짐 예약 | "연속" 또는 "H:MM" | 클릭 → 모달(4430) 열기 |
| 에러 상태 | "정상"(녹색) / 에러코드(빨강) | 표시 전용 |

각 행은 하나의 장치에 대응하며, 폴링에 의해 실시간으로 갱신된다.

#### 7.3 실시간 이벤트 패널 (참조번호 4300)

실시간 이벤트 패널(4300)은 화면 우측에 배치되며, 감지된 이벤트를 시간 역순으로 표시한다. 각 이벤트 항목은 시간, 장소, 장치 ID, 메시지로 구성된다. 이벤트이력 조회 버튼을 통해 이벤트이력 모달(4420)을 열 수 있으며, 이를 통해 저장된 로그 파일을 불러와 과거 이벤트를 조회할 수 있다.

#### 7.4 모달 다이얼로그

**(1) 통신설정 모달 (4400)**: 포트 경로, 전송 속도(9600~115200 bps), 데이터 비트, 패리티, 정지 비트, 흐름 제어 등의 통신 파라미터를 설정한다. 설정값은 localStorage에 영속적으로 저장된다. 입력값 검증 로직에 의해 허용 목록에 없는 값은 기본값으로 대체된다.

**(2) 환경설정 모달 (4410)**: 그룹별 장치 수(0~16대), 각 장치의 장소명을 설정한다. 최대 10개 그룹(0~9)을 지원하며, 설정값은 localStorage에 영속적으로 저장된다.

**(3) 이벤트이력 모달 (4420)**: 저장된 로그 파일을 불러와 과거 이벤트를 조회한다. 파일 선택 다이얼로그를 통해 로그 파일을 선택하고, 시간, 장소, ID, 메시지 컬럼으로 구성된 테이블에 이벤트 이력을 표시한다.

**(4) 꺼짐예약 모달 (4430)**: 특정 장치의 꺼짐 예약을 설정한다. 예약 활성/비활성 토글, 시간(0~24시), 분(0~59분)을 설정할 수 있다. 설정 시 sendOffReservation() 명령이 해당 장치로 전송된다.

### 8. 영속 설정 및 로그 관리

#### 8.1 통신 설정 영속화

통신 설정(CommSettings)은 다음의 필드를 포함하며, localStorage에 키 "dehumidifier-comm-settings"로 저장된다:

- path: COM 포트 경로 (문자열)
- baudRate: 전송 속도 (9600, 19200, 38400, 57600, 115200 중 택일)
- stopBits: 정지 비트 (1 또는 2)
- parity: 패리티 ("None", "Odd", "Even" 중 택일)
- flowControl: 흐름 제어 ("None", "Hardware", "Software" 중 택일)

로딩 시 입력값 검증 로직이 적용되어, 허용 목록에 없는 값은 기본값(baudRate: 9600, stopBits: 1, parity: "None", flowControl: "None")으로 대체된다.

#### 8.2 환경 설정 영속화

환경 설정(EnvSettings)은 그룹 번호(0~9)를 키로 하는 객체이며, 각 그룹은 장치 수(count, 0~16)와 장소명 배열(locationNames)을 포함한다. localStorage에 키 "dehumidifier-env-settings"로 저장된다. 그룹 뷰 선택 상태(0~9)는 별도의 키 "dehumidifier-view-group"으로 저장되어, 앱 재시작 후 동일 그룹 뷰를 복원한다.

#### 8.3 이벤트 로그 파일

이벤트 로그는 Electron 사용자 데이터 디렉토리(app.getPath("userData"))에 event.log 파일로 저장된다. TSV(Tab-Separated Values) 형식으로, 각 행은 "시간\t장소명\t장치ID\t메시지\n" 구조이다. fs.appendFileSync를 사용한 동기식 추가 기록 방식으로, 로그 누락을 방지한다.

### 9. 구현 예시

이하, 본 발명의 주요 알고리즘의 구현 예시를 제시한다.

#### 9.1 CRC-16 CCITT 계산 구현 예시

```
function crc16Ccitt(buffer, start, end):
    crc ← 0xFFFF
    for i ← start to end-1:
        crc ← crc XOR (buffer[i] << 8)
        for j ← 0 to 7:
            if (crc AND 0x8000) ≠ 0:
                crc ← ((crc << 1) XOR 0x1021) AND 0xFFFF
            else:
                crc ← (crc << 1) AND 0xFFFF
    return crc

function appendCrc16(buffer):
    crc ← crc16Ccitt(buffer, 0, buffer.length)
    out ← new array of (buffer.length + 2) bytes
    copy buffer to out
    out[buffer.length] ← (crc >> 8) AND 0xFF     // 상위 바이트
    out[buffer.length + 1] ← crc AND 0xFF         // 하위 바이트
    return out

function verifyCrc16(packet):
    if packet.length < 4: return false
    dataEnd ← packet.length - 2
    computed ← crc16Ccitt(packet, 0, dataEnd)
    received ← (packet[dataEnd] << 8) OR packet[dataEnd + 1]
    return computed = received
```

#### 9.2 ID 바이트 인코딩/디코딩 구현 예시

```
// 인코딩
function makeIdByte(groupId, deviceId):
    return ((groupId AND 0x0F) << 4) OR (deviceId AND 0x0F)

// 디코딩
function decodeIdByte(idByte):
    groupId ← (idByte >> 4) AND 0x0F
    deviceIndex ← idByte AND 0x0F
    return (groupId, deviceIndex)
```

#### 9.3 상태 요청 패킷 빌드 구현 예시

```
function buildStatusRequest(groupId, deviceId):
    buf ← new array of 3 bytes
    buf[0] ← 0x91               // CMD: 상태 요청
    buf[1] ← 0x01               // LEN: ID 1바이트
    buf[2] ← makeIdByte(groupId, deviceId)
    return appendCrc16(buf)      // CRC 부가 후 5바이트 반환
```

#### 9.4 에러 바이트 디코딩 구현 예시

```
ERROR_LABELS ← {0x01:"E1 고압", 0x02:"E2 저압", 0x04:"E3 에바동결",
                0x08:"E4 에바온도", 0x10:"E5 토출온도", 0x20:"E6 모듈온도",
                0x40:"E7 온습도", 0x80:"E8 만수위"}

function errorByteToStatus(errorByte):
    if errorByte = 0: return "정상"
    for each bit in [0x01, 0x02, 0x04, 0x08, 0x10, 0x20, 0x40, 0x80]:
        if (errorByte AND bit) ≠ 0:
            return ERROR_LABELS[bit]
    return "정상"

function errorByteToMessages(errorByte):
    messages ← []
    for each (bit, label) in ERROR_LABELS:
        if (errorByte AND bit) ≠ 0:
            messages.append(label)
    return messages
```

### 10. 변형예 및 응용

본 발명의 기술 사상은 상술한 실시예에 한정되지 않으며, 다음과 같은 변형예가 가능하다.

#### 10.1 프로토콜 확장

IProtocolAgent 공통 인터페이스(1200)를 구현하는 새로운 에이전트를 추가함으로써, Modbus RTU, BACnet, EtherNet/IP 등 다양한 산업용 프로토콜로의 확장이 가능하다. 이 경우, 프로토콜 추상화 계층(1300)의 registerAgent() 메서드를 통해 새로운 에이전트를 등록하기만 하면 되며, 상위 계층의 변경은 불필요하다.

#### 10.2 적용 장치 확장

본 실시예에서는 제습기를 대상 장치로 설명하였으나, 동일한 아키텍처를 공조기, 냉동기, 보일러, 펌프 등 다양한 산업용 장치의 모니터링에 적용할 수 있다. ParsedStatus 구조체와 DeviceRow 타입을 장치 특성에 맞게 확장함으로써 대응이 가능하다.

#### 10.3 폴링 주기 적응형 조절

고정 주기(1,500ms) 대신, 네트워크 부하, 장치 수, 에러 발생 빈도 등에 따라 폴링 주기를 적응적으로 조절하는 변형이 가능하다. 에러 발생 시 폴링 주기를 단축하여 빠른 상태 파악을 도모하고, 정상 상태에서는 주기를 연장하여 네트워크 부하를 저감할 수 있다.

#### 10.4 이벤트 감지 항목 확장

상술한 세 가지 이벤트 유형(압축기, 운전, 에러) 외에, 온도/습도의 급격한 변화, 설정값 변경, 예약 실행 등 추가적인 이벤트 유형을 정의하고 감지 조건을 확장할 수 있다.

#### 10.5 클라우드 연동

로컬 파일 시스템 기반의 이벤트 로깅 외에, 클라우드 데이터베이스나 메시지 큐로의 이벤트 전송을 추가하여, 원격 모니터링 및 분석을 수행할 수 있다.

*이상으로 본 발명의 바람직한 실시예에 대하여 상세히 설명하였으나, 본 발명의 기술적 사상은 상기 실시예에 한정되지 아니하며, 본 발명이 속하는 기술 분야에서 통상의 지식을 가진 자에 의해 다양한 변형이 가능함은 물론이다. 따라서, 본 발명의 권리 범위는 후술하는 청구범위에 의하여 해석되어야 하며, 그와 동등하거나 균등한 범위 내에 있는 모든 기술적 사상은 본 발명의 권리 범위에 포함되는 것으로 해석되어야 할 것이다.*

# 청구범위

## 에이전트 기반 멀티프로토콜 통신 모니터링 시스템 및 방법

---

### 독립항

**【청구항 1】**

산업용 장치(110)의 상태를 다중 통신 프로토콜을 통해 모니터링하는 에이전트 기반 멀티프로토콜 통신 모니터링 시스템에 있어서,

a) RS-485 통신(200) 및 UDP/TCP 네트워크 통신(500)을 포함하는 복수의 통신 프로토콜에 각각 대응하며, 접속(connect), 해제(disconnect), 폴링(poll), 명령 전송(sendCommand) 및 데이터 수신 콜백(onData)을 포함하는 공통 인터페이스(1200)를 구현하는 복수의 프로토콜 에이전트(910, 920, 930);

b) 상기 복수의 프로토콜 에이전트(910, 920, 930)를 등록하고, 각 프로토콜 에이전트로부터 수신된 장치 상태 데이터를 통합하여 프로토콜 종류에 무관한 통합 장치 상태(1400)를 상위 계층에 제공하는 프로토콜 추상화 계층(1000, 1300);

c) 설정 가능한 주기로 상기 프로토콜 추상화 계층(1300)을 통해 각 장치(110)에 상태 요청 패킷을 순차적으로 전송하는 폴링 엔진(2000);

d) 이전 폴링 주기에서 수집된 장치 상태와 현재 폴링 주기에서 수집된 장치 상태를 비교하여, 상태 변화가 발생한 항목을 검출하는 상태 변화 감지부(2600);

e) 상기 상태 변화 감지부(2600)에 의해 상태 변화가 검출된 경우 이벤트를 생성하고(3200), 이벤트 발생 시 제1 음향 신호를 출력하는 이벤트음 재생 경로(3330)와, 에러 상태 지속 시 제2 음향 신호를 주기적으로 반복 출력하는 경고음 루프 제어부(3400)를 포함하는 이중 음향 경보부; 및

f) 상기 통합 장치 상태(1400) 및 생성된 이벤트를 사용자에게 표시하며, 장치 제어 테이블(4200)과 실시간 이벤트 패널(4300)을 포함하는 대시보드 표시 계층(1100, 4000);

을 포함하는 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 시스템.

---

**【청구항 2】**

다중 통신 프로토콜을 통해 산업용 장치(110)를 모니터링하는 방법에 있어서,

a) RS-485 에이전트(910), UDP 에이전트(920) 및 TCP 에이전트(930)를 포함하는 복수의 프로토콜 에이전트가 접속(connect), 해제(disconnect), 폴링(poll), 명령 전송(sendCommand) 및 데이터 수신 콜백(onData)을 포함하는 공통 인터페이스(1200)를 통해 서로 다른 통신 프로토콜을 추상화하는 단계;

b) 그룹 식별자와 장치 인덱스로 구성된 계층적 주소 체계를 이용하여, 소정의 주기(2000)마다 선택된 그룹에 속한 각 장치(110)에 대해 상태 요청 패킷을 순차적으로 전송하는 폴링 단계;

c) 상기 폴링 단계에서 수신된 데이터를 스트림 버퍼(rxBuffer)에 누적하고, 누적된 데이터로부터 소정의 프레임 시작 패턴을 탐색하여 완전한 프레임을 추출하는 프레임 추출 단계(2300, 2400);

d) 추출된 프레임에 대해 CRC-16 검증을 수행하여 프레임 무결성을 확인하고, 검증에 실패한 프레임을 폐기하는 무결성 검증 단계;

e) 검증된 프레임으로부터 장치 상태를 파싱하고, 이전 폴링 주기에서 수집된 장치 상태(previousStatus)와 현재 파싱된 장치 상태(currentParsedStatus)를 비교하여, 상태 변화가 발생한 항목을 검출하는 상태 변화 감지 단계(2500, 2600); 및

f) 상기 상태 변화가 검출된 경우, 이벤트를 생성하고(3200), 이벤트 발생 시 제1 주파수의 단발성 이벤트음을 출력하고(3330), 에러 상태가 존재하는 경우 제2 주파수의 경고음을 소정의 간격으로 반복 출력하는(3400) 이중 음향 경보 단계;

를 포함하는 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 방법.

---

**【청구항 3】**

직렬 통신에서 단편화되어 수신되는 데이터를 처리하는 방법에 있어서,

a) 통신 포트(610)로부터 수신된 바이트 단편(chunk)을 수신 버퍼(rxBuffer)에 순차적으로 누적하는 바이트 누적 단계(2300);

b) 상기 수신 버퍼(rxBuffer) 내에서 응답 명령 코드와 데이터 길이 바이트로 구성된 프레임 시작 패턴을 탐색하는 패턴 탐색 단계(2400);

c) 상기 프레임 시작 패턴이 발견된 위치로부터 소정의 고정 길이만큼의 바이트를 슬라이스하여 완전한 프레임을 추출하는 프레임 추출 단계;

d) 추출된 프레임의 명령 바이트부터 마지막 데이터 바이트까지에 대하여 CRC-16 값을 연산하고, 연산된 CRC-16 값과 프레임에 포함된 CRC-16 값을 비교하여 무결성을 검증하는 CRC 검증 단계;

e) 무결성이 검증된 프레임으로부터 ID 바이트(1520)의 상위 니블 및 하위 니블을 역매핑하여 그룹 식별자 및 장치 인덱스를 결정하고, 데이터 바이트들을 비트 필드(1530) 단위로 파싱하여 장치 상태(1400)를 생성하는 상태 파싱 단계(2500); 및

f) 상기 수신 버퍼(rxBuffer)에서 처리가 완료된 바이트를 제거하고 미처리 잔여 바이트를 유지하는 버퍼 정리 단계(2700);

를 포함하는 것을 특징으로 하는, 직렬 통신 데이터 처리 방법.

---

**【청구항 4】**

제2항에 기재된 에이전트 기반 멀티프로토콜 통신 모니터링 방법을 컴퓨터에서 실행시키기 위한 프로그램을 기록한 컴퓨터 판독 가능 기록매체.

### 종속항

**【청구항 5】**

제1항에 있어서,

상기 복수의 프로토콜 에이전트(910, 920, 930)는 장치(110)의 주소를 1바이트 ID(1520)로 인코딩하되, 상기 1바이트 ID의 상위 4비트(bit[7:4])를 그룹 식별자(GroupID, 0~9)로 할당하고 하위 4비트(bit[3:0])를 그룹 내 장치 인덱스(DeviceIndex, 0~15)로 할당하여, 최대 160개의 장치를 고유하게 지정하는 그룹-장치 주소 체계를 사용하는 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 시스템.

---

**【청구항 6】**

제1항에 있어서,

상기 프로토콜 추상화 계층(1000, 1300)은 IProtocolAgent 공통 인터페이스(1200)를 정의하되, 상기 IProtocolAgent 공통 인터페이스(1200)는,

- 통신 연결 설정을 수행하는 connect() 메서드;
- 통신 연결 해제를 수행하는 disconnect() 메서드;
- 지정된 장치 목록에 대해 상태 요청을 수행하는 poll() 메서드;
- 특정 그룹 식별자 및 장치 인덱스로 명령을 전송하는 sendCommand() 메서드; 및
- 수신 데이터에 대한 콜백 함수를 등록하는 onData() 메서드; 및
- 에이전트의 프로토콜 유형을 반환하는 getProtocolType() 메서드;

를 포함하고, 상기 RS-485 에이전트(1210), UDP 에이전트(1220) 및 TCP 에이전트(1230)가 각각 상기 IProtocolAgent 공통 인터페이스(1200)를 구현하는 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 시스템.

---

**【청구항 7】**

제1항에 있어서,

상기 시스템은 하드웨어 계층(600)에 접근하는 메인 프로세스(700)와, 샌드박스화된 렌더러 프로세스(1100) 사이에 IPC 보안 격리 경계(800)를 더 포함하되, 상기 IPC 보안 격리 경계(800)는 contextBridge를 통해 미리 정의된 API만을 렌더러 프로세스(1100)에 노출하고, ipcRenderer.invoke 및 ipcMain.handle 방식의 양방향 통신을 통해 프로세스 간 데이터를 전달하는 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 시스템.

---

**【청구항 8】**

제1항에 있어서,

상기 이중 음향 경보부는,

- 이벤트 발생 시 약 800Hz의 제1 주파수로 약 100ms 동안 단발성 음향 신호를 출력하는 이벤트음 재생 경로(3330); 및
- 에러 상태가 지속되는 동안 약 600Hz의 제2 주파수로 약 150ms 동안의 음향 신호를 약 1초 간격으로 반복 출력하는 경고음 루프 제어부(3400);

를 포함하는 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 시스템.

---

**【청구항 9】**

제2항에 있어서,

상기 폴링 단계에서, 선택된 그룹에 속한 각 장치(110)에 대하여 소정의 폴링 주기(2000)마다 상태 요청 패킷(1500)을 순차적으로 전송하되, 상기 소정의 폴링 주기는 1500밀리초인 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 방법.

---

**【청구항 10】**

제2항에 있어서,

상기 상태 변화 감지 단계에서, 적어도,

- 압축기(compressor) 동작 상태의 켜짐/꺼짐 변화;
- 운전 시작(runStart) 상태의 시작/정지 변화; 및
- 에러 코드(errorCode)의 발생/해제 변화;

를 포함하는 3종의 상태 변화 유형을 검출하는 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 방법.

---

**【청구항 11】**

제2항에 있어서,

상기 이중 음향 경보 단계 이후에, 검출된 이벤트를 탭(Tab) 구분 형식의 로그 문자열로 변환하고, 파일 시스템의 로그 파일(3320)에 추가 기록(appendFileSync)하여 영구적으로 저장하는 이벤트 로깅 단계를 더 포함하는 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 방법.

---

**【청구항 12】**

제3항에 있어서,

상기 프레임 시작 패턴은 응답 명령 코드(0xC1)와 이에 연속하는 데이터 길이 바이트(0x0D)로 구성되며(1510), 상기 소정의 고정 길이는 18바이트인 것을 특징으로 하는, 직렬 통신 데이터 처리 방법.

---

**【청구항 13】**

제3항에 있어서,

상기 CRC 검증 단계에서, CRC-16 CCITT 다항식(0x1021)과 초기값(0xFFFF)을 사용하여 프레임의 명령 바이트부터 마지막 데이터 바이트까지에 대해 CRC-16 값을 연산하고, 연산된 2바이트 CRC 값을 빅엔디안(big-endian) 순서로 프레임 말미에 부가된 CRC 바이트와 비교하는 것을 특징으로 하는, 직렬 통신 데이터 처리 방법.

---

**【청구항 14】**

제1항에 있어서,

상기 시스템은 통신 파라미터 및 환경 설정 값을 로컬 저장소(localStorage)에 저장하고, 애플리케이션 재시작 시 상기 로컬 저장소로부터 저장된 통신 파라미터 및 환경 설정 값을 복원하는 영구 설정 저장부(4400, 4410)를 더 포함하는 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 시스템.

---

**【청구항 15】**

제2항에 있어서,

상기 폴링 단계에서의 계층적 주소 체계는, 단일 주소 바이트(1520)의 상위 니블(bit[7:4])에 그룹 식별자를 인코딩하고 하위 니블(bit[3:0])에 장치 인덱스를 인코딩하되, flatIndex = groupId x 16 + deviceIndex 수식에 의해 최대 160개의 장치를 고유하게 지정하는 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 방법.

---

**【청구항 16】**

제8항에 있어서,

상기 경고음 루프 제어부(3400)는 모니터링 대상 장치(110) 중 어느 하나의 장치라도 에러 상태(errorCode != 0)를 보고하는 경우 경고음 반복 출력을 개시하고(hasAnyDeviceError = true), 모든 장치의 에러 상태가 해소된 경우(hasAnyDeviceError = false) 경고음 반복 출력을 자동으로 중지하는 것을 특징으로 하는, 에이전트 기반 멀티프로토콜 통신 모니터링 시스템.
